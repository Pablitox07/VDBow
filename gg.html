I have found a solution to the problem.  Changing the version number of the package that fails to load from 8.0.3.0 to 8.0.6.0 solved the problem as detailed in below in a message to our team. It seems that Azure .net runtime is altering the package reference when it trying to load at runtime the Microsoft.Extensions.Logging.AzureAppServices module.  Version 8.0.3.0 compiles and runs in the local dev environment. Why does this happen when running in the cloud?? Azure wants version=8.0.6.0 and in the project files we have 8.0.3.0.  This requirement supercedes what you have in the project file. .Net assembles have their version numbers embedded in the file and are not in the file name.  So yes we had a file named Microsoft.Extensions.Logging.AzureAppServices, but it was not 8.0.6.0. Solution, change project file to request 8.0.6.0. Not fun +++++
